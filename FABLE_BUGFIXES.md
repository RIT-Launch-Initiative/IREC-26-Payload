# Fable Branch Bug Fixes

- [cubesat_captain/cubesat_comms/lora.c] unpack_lora_link_change masked frequency bytes AFTER shifting (`(buf[2] << 24) & 0xff` == 0), so any over-the-air link change tuned the radio to a garbage frequency built from only the low byte; fixed mask-then-shift.
- [cubesat_captain/cubesat_comms/packets_g2p.c] unpack_uint32 checked `len < 2` but reads 4 bytes — out-of-bounds read on short packets; fixed to `len < 4`.
- [cubesat_captain/cubesat_comms/packets_g2p.c] pack_phototransform masked output_width to 11 bits while unpack_phototransform reads 13 bits; widened pack mask to 13 bits to match.
- [cubesat_captain/captain_node.cpp] requestStateChange returned `success = true` for an invalid state; now returns false with the reason.
- [cubesat_captain/captain_node.cpp] expert_for_state bounds check used `>` so `state == NumStates` indexed one past the end of the experts array; fixed to `>=`.
- [cubesat_captain/captain_node.cpp] setCamera dereferenced camera_gpio without a null check (it is null whenever openCameraLine fails); now guarded and the gpiod return code is propagated.
- [cubesat_captain/captain_node.cpp] restart_system() is an unimplemented stub — the payload can flag for a new flight dir but can never actually restart; implemented on feature/flight-dir-rotation.
- [cubesat_captain/captain_comms.cpp] Command_SendArmTargetForPhotoAndComeBack dispatched to send_arm_to_target_and_come_back (no photo); now calls send_arm_to_target_and_come_back_with_photo.
- [cubesat_captain/captain_comms.cpp] load_block_data returned `false` (which converts to 0 == success) on exception, so missing/corrupt image blocks were transmitted as uninitialized garbage; now returns -1.
- [cubesat_captain/status_accumulator.hpp] last_image_id was uninitialized and fed into std::max on the first image metadata update; initialized to 0.
- [cubesat_captain/pad_expert.*] boost detection result has_boosted_ was write-only — nothing ever transitioned Preboost to Flight, so an autonomous launch would never be detected; PreboostExpert now enters Flight when boost is detected, and the threshold is compared against magnitude in excess of 1 g (raw magnitude on the pad is ~9.81 m/s², which already exceeded the 7 m/s² threshold). Boost state and the moving average reset on Pad/Preboost entry.
- [cubesat_captain/pad_expert.hpp] CCircularBuffer::NewestSample computed `oldest_index - 1` on an unsigned index with a dead `< 0` check — out-of-bounds read at index SIZE_MAX when oldest_index is 0; fixed with modular arithmetic.
- [cubesat_captain/flipping_expert.cpp] `port` face vector was (0,-1,0), identical to `starboard`, so which_side could never identify the single-wall orientation and the wrong flip servo could fire; corrected to (0,1,0).
- [cubesat_captain/flipping_expert.cpp] enter_state never reset attempts/opened_side flags, so re-entering Flipping after a previous run could immediately escalate to Emergency; reset them on entry.
- [cubesat_captain/autonomous_experts.cpp] decide_next finished the arm path at `size()-1`, skipping the photo at the final pose, while the post-photo thread checked `path_index > size()` allowing an out-of-bounds path read; both off-by-ones fixed to `>= size()`.
- [cubesat_captain/autonomous_experts.cpp] the detached 2 s camera-wait thread continued driving the arm even if the captain had left the state (manual override / emergency) while it slept; it now rechecks the active state before continuing the path.
- [cubesat_captain/packet_writer.cpp] battery clamping compared volts against millivolt limits (±32768 V) and cast through uint16_t; now clamps the millivolt value and casts to int16_t.
- [cubesat_pi_io/pi_io_node.cpp] buzzer.gpio_chip default was "/dev/gpiochip0" but gpiod_chip_open_by_name takes a bare name; default fixed to "gpiochip0".
- [cubesat_pi_io/buzzer_driver.hpp] `int repeat_count = false;` — wrong type and bool initializer for a count assigned from uint32_t; now `uint32_t repeat_count = 0`.
- [cubesat_stm_bridge/crashout_stm.cpp] CrashoutSTM::open never stored speed_hz, so every SPI transfer used an uninitialized member for spi_ioc_transfer.speed_hz (undefined behavior); now stored in open() and the member has a sane default.
- [cubesat_stm_bridge/crashout_stm.cpp] getServoMotion took subspan(2, 7) of an 8-byte transfer (reads past the end, UB); payload is 5 bytes, fixed to subspan(2, 5).
- [cubesat_stm_bridge/stm_bridge_node.cpp] normed_v16 divided by a zero norm before the first IMU sample arrived (status timer runs at 50 Hz from construction), producing NaN and UB on the int16 cast; zero vector is now returned for zero/non-finite norms.
- [cubesat_stm_bridge/stm_bridge_node.cpp] onStatusTimer returned early when the STM status read failed, so a persistent SPI failure mid-action meant the accepted goal never got a result (client hangs forever); the mode tick now still runs on stale status so the action timeout can abort it.
- [cubesat_stm_bridge/stm_bridge_node.cpp] holdShut left the response unset when asked to release a hold while in a non-Idle, non-Holding mode; now explicitly declines.
- [cubesat_radio/radio_node.cpp] linkTestChanceExpired contained copy-pasted lines from signalStopping that stopped the interrupt thread — 30 s after any link-test window expired the radio permanently stopped receiving; removed.
- [cubesat_radio/radio_node.cpp] radioLoop read the event flags with load() then store(0), losing any flag set between the two operations (missed interrupts/packets); now uses exchange(0).
- [cubesat_radio/radio_node.cpp] a transient setReceiveMode failure after an immediate reconfigure did `return`, silently killing the radio thread for the rest of the flight; now logs and keeps running.
- [cubesat_radio/radio_node.cpp] submitPacketToSend accepted empty (and >255 byte) packets; an empty packet would index packet[0] in put_queue_length_to_header (UB) and an oversized one is unsendable; both rejected at the queue.
- [cubesat_radio/sx1262_radio.cpp] the send() failure path set the RF switch to TX (copy-paste) and the success path never set it back to RX, leaving the external RF switch in the TX position while in receive mode; both paths now return the switch to RX. send() also now reports TX-done timeout as failure.
- [cubesat_radio/sx1262_radio.cpp] wait_for_irq returned true on timeout; returns false so timeouts are distinguishable from IRQ delivery.
- [cubesat_watch/ImageHandlerNode.cpp] the constructor scanned flight_dir instead of the images subdirectory for the next image id, so the on-boot republish of the latest image metadata never matched what handleImageRequest writes; now scans saveDirectory.
- [cubesat_watch/ImageTaker.cpp] mmap failure was compared against NULL instead of MAP_FAILED, so a failed mapping was used as a frame buffer.
- [cubesat_watch/ImageTaker.cpp] take_image_and_crop spun forever waiting for the camera request to complete — a wedged camera (seen when transmitting at high power) hung the watch node permanently; now times out after 10 s and returns an empty Mat.
- [cubesat_watch/ImageTaker.cpp] crop/downscale parameters arriving over the radio were not validated; an out-of-range crop rect or zero output width made OpenCV throw in the libcamera completion handler. Values are now clamped with a full-frame fallback.
- [cubesat_bringup/flight.launch.py] ros2 bag record was passed '--compression-mode file' as a single argument (and the -x regex was shell-quoted); split into separate arguments and dropped shell=True.
- [cubesat_bringup/scripts/start_flight.py] script crashed on every run: NameError on undefined `path` and `next_image_id`, `os.exists` (doesn't exist), unexpanded `~` in the base path, and it passed a `next_image_id` launch argument that flight.launch.py doesn't declare; rewritten to expand the base dir, create it if needed, and launch with just flight_dir.
- [cubesat_captain/CMakeLists.txt, cubesat_stm_bridge/CMakeLists.txt, cubesat_radio/CMakeLists.txt] all three installed a `config` directory that does not exist in their package (configs live in cubesat_bringup), so `colcon build` failed at the install step for every one of them; removed the bogus install stanzas.
- [cubesat_watch/CMakeLists.txt] libcamera include path was hardcoded to `${rclcpp_DIR}/../../../include/libcamera/`, which only exists on the Pi image; now discovered via pkg-config with the ROS-prefix path as fallback.
